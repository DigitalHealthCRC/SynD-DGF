Five-step synthetic data governance framework

The SynD-DGF provides a structured approach to synthetic health data governance through five sequential steps that ensure privacy compliance, technical quality, and regulatory alignment.

Guiding principles

  • Purpose limitation: Synthetic data generation must be justified by a clear public benefit purpose
  • Data minimisation: Only the minimum data necessary should be used and shared
  • Transparency: Clear documentation and accountability throughout the process
  • Privacy by design: Privacy considerations embedded at every step
  • Technical robustness: High-quality synthetic data that preserves utility while protecting privacy
  • Continuous assessment: Ongoing evaluation of privacy risks and data quality

The five-step process

Each step builds upon the previous one, with clear decision points and assessment criteria:

1

Assess use case

Purpose: Determine if synthetic data generation is appropriate and lawful for your intended use case.

Key activities:

  • Evaluate public benefit purpose and consumer health outcomes
  • Assess community and stakeholder notifications
  • Complete use case assessment checklist (3 required tests)
  • Conduct impact assessment including data ethics and Indigenous data sovereignty

Decision point: Proceed only if use case meets public benefit criteria and passes all required assessments.

View Step 1 Details
2

Assess and prepare source data

Purpose: Ensure source data is suitable, available, and properly prepared for synthetic data generation.

Key activities:

  • Identify and validate data availability and access rights
  • Complete technical assessment for data quality and fitness for purpose
  • Remove or augment directly identifying information
  • Prepare data specification and quality statements

Decision point: Proceed only if source data is fit for purpose and properly controlled by your organisation.

View Step 2 Details
3

Generate synthetic data

Purpose: Create synthetic data using appropriate models and techniques whilst maintaining data quality and utility.

Key activities:

  • Select appropriate generative model (rule-based, statistical, or machine learning)
  • Configure model parameters and training procedures
  • Generate synthetic dataset maintaining statistical properties
  • Document model details and generation parameters

Decision point: Proceed only if synthetic data maintains utility whilst reducing privacy risks.

View Step 3 Details
4

Assess re-identification risks

Purpose: Evaluate and mitigate the risk that synthetic data could be used to re-identify individuals.

Key activities:

  • Apply appropriate de-identification techniques (K-anonymity, differential privacy)
  • Conduct re-identification risk assessment using established methodologies
  • Test for data leakage and membership inference risks
  • Apply additional privacy-preserving measures if needed

Decision point: Proceed only if re-identification risk is assessed as very low.

View Step 4 Details
5

Manage residual privacy risks

Purpose: Implement safeguards and controls to manage any remaining privacy risks during data use and sharing.

Key activities:

  • Apply Five Safes Framework (safe data, projects, settings, people, outputs)
  • Establish data sharing agreements and use agreements
  • Implement security controls and access management
  • Plan ongoing assurance and compliance monitoring

Final decision: Approve data sharing only if all safeguards are in place and risks are acceptable.

Roles and responsibilities

Data Provider

  • Owns and controls the source dataset
  • Responsible for data quality and compliance
  • Conducts privacy and technical assessments
  • Makes final decision on data sharing approval

Data Requestor

  • Defines the use case and data requirements
  • Ensures appropriate technical capabilities
  • Complies with data sharing agreements
  • Manages end user access and training

Complex scenarios

Some synthetic data projects may be classified as 'complex' and require additional assessment:

  • Cross-organisational data: When source data is held by multiple organisations
  • Third-party generation: When synthetic data generation is outsourced
  • High re-identification risk: When risk assessment indicates more than very low risk
  • Sensitive populations: When data involves Indigenous communities or vulnerable groups

For complex scenarios, consult the decision tree navigator for additional guidance.

Important note
