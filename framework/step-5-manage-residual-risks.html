Step 5: Manage residual privacy risks

Even with comprehensive de-identification, synthetic data may still carry residual privacy risks. This final step implements robust safeguards to ensure safe use and sharing.

Step objectives

  • Apply the Five Safes Framework to manage residual privacy risks
  • Establish comprehensive data sharing agreements and use agreements
  • Implement appropriate security controls and access management
  • Plan ongoing assurance and compliance monitoring activities
  • Make final approval decision for synthetic data sharing

Five Safes Framework for synthetic data

The Five Safes Framework provides a structured approach to managing two key types of privacy risks: inadvertent disclosure (re-identification) and misuse of data by authorised users.

Risk balancing principle

The five safes work together to create an overall level of safety. Different elements can balance each other - if one control is "dialled up", it may be safe to "dial down" another control.

Example: When releasing data to the public, there is zero safety in terms of 'people' or 'settings'. To achieve overall safety, you must 'dial up' other elements like data safety through extremely stringent de-identification.

Safe data

Question: Has appropriate and sufficient protection been applied to the data?

Key considerations

  • Data presentation: Is data clearly understood, appropriately footnoted, with sources acknowledged?
  • Data integrity: Will data quality be maintained by the Data Requestor?
  • Data lineage: Will data remain segregated with clear provenance maintained?
  • Re-identification testing: Has re-identification risk been comprehensively tested?

Potential risk factors

High-risk data fields
  • Date of birth or precise age
  • Combinations of demographic fields (age, gender, postcode, ethnicity)
  • Attribute data that could uniquely identify individuals
  • Geolocation or longitudinal data
  • Text-based or unstructured data fields
High protection (dial up)
  • Strong differential privacy guarantees
  • Extensive suppression and generalisation
  • Multiple independent privacy audits
  • Formal privacy-preserving guarantees
Standard protection
  • K-anonymity with appropriate k values
  • Standard de-identification techniques
  • Comprehensive documentation and metadata
  • Regular data quality assessments

Safe projects

Question: Is the data to be used for an appropriate purpose?

Purpose assessment

  • Privacy Impact Assessment: Has a comprehensive PIA been completed?
  • Public benefit purpose: Does the use clearly benefit consumer health or health system outcomes?
  • Data ownership clarity: Are data ownership and control arrangements clear?
  • Methodology appropriateness: Is synthetic data suitable for the research methodology?

Reasons for concern

  • Data owner outside the participating organisations
  • Data was indirectly collected, generated, or inferred
  • New metrics or indicators need to be built for this proposal
  • Synthetic data request lacks clear objective or methodology
High scrutiny (dial up)
  • Independent ethics review
  • External advisory committee oversight
  • Public consultation on proposed use
  • Regular purpose limitation audits
Standard oversight
  • Internal ethics and governance review
  • Clear purpose limitation clauses
  • Regular project milestone reviews
  • Outcome reporting requirements

Safe settings

Question: Does the access environment prevent unauthorised use?

Infrastructure requirements

  • Data Requestor capability: Can they meet all Data Provider security requirements?
  • Secure transfer and storage: Are secure mechanisms in place for data handling?
  • Technical security assessment: Has the system undergone appropriate security evaluation?
  • Data classification: Is data labelled according to classification policies?

Required controls

Legal frameworks
  • Legally binding Data Sharing Agreement (DSA)
  • Data Use Agreements (DUA) with end users
  • Appropriate confidentiality and privacy provisions
  • Clear data return or disposal obligations
Technical controls
  • User access controls and monitoring
  • Data encryption in transit and at rest
  • Audit trails and access logging
  • System security certifications

Storage location requirements

Australian requirement: Data must be held in Australia by an organisation subject to legal privacy obligations and oversight by a statutory regulator (Privacy Commissioner or Information Commissioner).

Note: Private sector organisations with annual turnover <$3 million and South Australian public sector agencies are not subject to legal privacy obligations. Any exception requires legal risk assessment and appropriate contractual clauses.

High security (dial up)
  • Secure research environment access only
  • Multi-factor authentication required
  • Real-time access monitoring and alerts
  • Data download restrictions
Standard security
  • Standard organisational security controls
  • Regular access reviews and audits
  • Data sharing agreement compliance
  • Incident response procedures

Safe people

Question: Is the requestor or end user appropriately authorised to access and use the data?

User authorisation

  • Approved user list: Has the Data Requestor provided a list of authorised users?
  • Specialised skills assessment: Do users possess skills that may increase re-identification risk?
  • Access to additional data: Do users have access to relevant datasets that could enable linkage?
  • Training requirements: Have users completed required privacy and security training?

Risk factors

  • Advanced technical skills in data linkage or re-identification
  • Access to complementary datasets from same population
  • Previous involvement in data collection for source dataset
  • Insufficient training in privacy-preserving data analysis
High trust requirements (dial up)
  • Individual background checks
  • Specialised privacy training certification
  • Regular competency assessments
  • Individual Data Use Agreements
Standard authorisation
  • Organisational authorisation
  • Standard privacy and security training
  • Role-based access controls
  • Regular access reviews

Safe outputs

Question: Are the statistical results non-disclosive?

Output management

  • Publication plans: How will the Data Requestor use or publish data and analysis outputs?
  • Re-identification guarantee: What assurance exists that outputs won't lead to re-identification?
  • Community harm prevention: Are there safeguards against harm to cohorts or communities?
  • Data Provider attribution: Will outputs identify data as originating from the Data Provider?

Output review requirements

  • All outputs reviewed before publication or sharing
  • Statistical disclosure control applied to results
  • Small cell suppression for aggregated statistics
  • Prohibition on attempts to re-identify individuals
High output control (dial up)
  • All outputs require Data Provider approval
  • Independent statistical disclosure review
  • Embargo periods for sensitive findings
  • Community consultation on findings
Standard output control
  • Standard statistical disclosure control
  • Output review by authorised personnel
  • Publication notification requirements
  • Citation and attribution requirements

Data sharing and use agreements

Comprehensive legal agreements are essential for managing residual privacy risks and ensuring compliance with privacy obligations.

Data Sharing Agreement (DSA)

A legally binding agreement between the Data Provider and Data Requestor organisations covering all aspects of synthetic data sharing.

Essential elements

  • Parties identification: Clear identification of all organisations involved
  • Data description: Detailed description of synthetic data including source dataset
  • Approved purposes: Specific, limited purposes for data use
  • Access restrictions: Limitations on who may access data
  • Security requirements: Technical and organisational security measures
  • Confidentiality obligations: Protection of data and any derived information
  • Retention and disposal: Data retention periods and destruction requirements
  • Compliance monitoring: Audit rights and compliance verification

Specific synthetic data clauses

  • Re-identification prohibition: Explicit prohibition on attempts to re-identify individuals
  • Combination restrictions: Limitations on combining synthetic data with other datasets
  • Model access restrictions: Prohibitions on accessing or reverse-engineering generation models
  • Quality disclaimers: Clear statements about synthetic data limitations

Privacy compliance clauses

  • Compliance with Australian Privacy Principles and relevant state/territory privacy laws
  • Incident reporting and breach notification procedures
  • Data subject rights and access request procedures
  • Cross-border data transfer restrictions and safeguards

Data Use Agreement (DUA)

Individual agreements completed by end users who will access the synthetic dataset, typically less detailed than the overarching DSA.

Essential elements for end users

  • User acknowledgment: Declaration of understanding responsibilities and obligations
  • Purpose limitations: Agreement to use data only for approved purposes
  • Access restrictions: Commitment to access data only during approved timeframes
  • Confidentiality: Protection of data and restriction on unauthorised sharing
  • Security compliance: Adherence to organisational security policies
  • Incident reporting: Obligation to report any privacy or security incidents
  • Training requirements: Completion of required privacy and security training

Output control provisions

  • Restrictions on publishing or sharing analysis outputs
  • Requirements for output review and approval processes
  • Statistical disclosure control obligations
  • Attribution and citation requirements

Ongoing assurance and monitoring

Once synthetic data is approved for sharing, implement ongoing activities to ensure compliance and address any issues.

Technical assurance

  • Security certification review: Regular review of security certifications and system specifications
  • System validation: Confirming data storage in approved systems with effective IT controls
  • Access management audits: Periodic review of user access and timely revocation procedures
  • Data access monitoring: Regular review of access logs for unusual behaviour or unauthorised access
  • Technical security assessments: Ongoing evaluation of system security measures

Compliance assurance

  • Training verification: Ensuring required privacy and security training is current
  • Output review: Checking analysis outputs align with approved use cases
  • Agreement compliance: Auditing compliance with Data Sharing Agreement terms
  • Third-party management: Monitoring service provider arrangements and performance
  • Policy compliance: Regular assessment against organisational policies

Risk management

  • Privacy impact monitoring: Ongoing assessment of privacy risks and mitigation effectiveness
  • Incident response: Procedures for responding to privacy or security incidents
  • Risk reassessment: Periodic review of re-identification and other privacy risks
  • Stakeholder feedback: Collecting and responding to community and user feedback
  • Regulatory compliance: Monitoring changes in privacy laws and regulatory requirements

Final approval decision

The accountable decision-maker must assess all safeguards and make the final determination on synthetic data sharing approval.

Decision criteria

✓ Approve synthetic data sharing

All safeguards are in place and residual risks are acceptable. The synthetic data may be shared according to the agreed terms and conditions.

Final steps before data release

  • Obtain formal approval signature from accountable decision-maker
  • Execute Data Sharing Agreement between organisations
  • Complete Data Use Agreements with all end users
  • Implement all technical and administrative controls
  • Begin ongoing monitoring and assurance activities

⚠️ Conditional approval with additional safeguards

Synthetic data sharing may proceed with enhanced controls and monitoring:

  • Enhanced monitoring: More frequent audits and compliance checks
  • Restricted access: Additional limitations on user access and data use
  • Output controls: Mandatory review of all outputs before release
  • Time limitations: Reduced data retention periods and regular reviews
  • Specific conditions: Any additional requirements identified during assessment

✗ Do not approve synthetic data sharing

If safeguards are insufficient or risks remain too high, the synthetic data should not be shared. Consider:

  • Additional privacy-preserving measures and re-assessment
  • Restricted internal use only within the Data Provider organisation
  • Alternative approaches such as aggregated statistics or traditional de-identification
  • Addressing identified gaps in safeguards before reconsidering approval

Framework completion

You have successfully completed the SynD-DGF five-step framework. Your synthetic health data project now has:

  • ✓ Confirmed public benefit purpose and stakeholder approval
  • ✓ High-quality source data properly prepared and documented
  • ✓ Appropriately generated synthetic data with preserved utility
  • ✓ Comprehensive privacy risk assessment and mitigation
  • ✓ Robust safeguards for ongoing data protection

Common scenarios and guidance

Situation: Releasing synthetic data publicly with no access controls.

Guidance:

  • Dial up "Safe Data" to maximum - stringent de-identification required
  • Zero control over "Safe People" or "Safe Settings" - cannot rely on these safeguards
  • "Safe Projects" control through documentation and license terms only
  • "Safe Outputs" cannot be controlled - users can publish anything
  • Consider if public release is appropriate given these constraints

Key consideration: Public release requires extremely high confidence in privacy protection through de-identification alone.

Situation: Sharing synthetic data with external research partners or collaborating organisations.

Guidance:

  • Comprehensive Data Sharing Agreement essential
  • Individual Data Use Agreements for all end users
  • Verify partner organisation security capabilities
  • Establish clear output review and approval processes
  • Regular compliance monitoring and audit rights

Related tool: Legal Pathways Wizard

Situation: Using cloud services for synthetic data storage or analysis.

Guidance:

  • Ensure cloud provider meets Australian data sovereignty requirements
  • Verify provider is subject to privacy law oversight (Privacy Commissioner)
  • Implement encryption for data at rest and in transit
  • Establish clear data processing agreements with cloud provider
  • Consider whether cloud processing increases re-identification risks

Important: Data must remain in Australia with organisations subject to privacy oversight.

Situation: Deciding which safes to "dial up" or "dial down" based on your specific use case.

Guidance:

  • Start by identifying which safes you have limited control over
  • Compensate by strengthening safes where you have more control
  • Document the rationale for your balancing decisions
  • Consider stakeholder expectations and risk tolerance
  • Review the balance as circumstances change

Example: If sharing with external researchers ("Safe People" lower), strengthen "Safe Settings" (secure research environment) and "Safe Outputs" (mandatory review).
